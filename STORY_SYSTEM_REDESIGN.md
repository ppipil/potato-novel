# 小说系统重设计方案

这份文档不是解释当前实现，而是重新设计一版更容易理解、也更适合继续迭代的小说系统。

目标很明确：

- 模板故事不要在游玩过程中再调模型
- 自定义故事可以生成，但生成链路要和模板故事彻底分开
- 前端负责播放和交互，后端负责内容资产和进度管理
- 把现在混在一起的“模板 / 自定义 / hydrate / 缓存 / 运行时生成”拆开

---

## 1. 先统一名词

### 1.1 `opening`

`opening` 就是故事开头。

它有两种来源：

- **模板 opening**
  - 系统预设好的故事开头
  - 来自书架里的模板卡片
  - 所有用户看到的是同一批 opening

- **自定义 opening**
  - 用户自己输入的故事开头
  - 每次输入都可能不同

### 1.2 `story package`

`story package` 是一整份可玩的互动小说内容包。

它至少包含：

- 标题
- 根节点
- 全部节点
- 每个节点的正文
- 每个节点的选项
- 节点之间的跳转关系
- 结局节点
- 初始状态

一句话：

**package 是“内容本体”。**

### 1.3 `session`

`session` 不是故事本体，而是“某个用户这次玩到了哪里”。

它至少记录：

- 当前用户是谁
- 正在玩的 package 是哪一份
- 当前节点 id
- 已经走过的 path
- 当前 state
- 是否完成
- 最终 completed run

一句话：

**session 是“游玩进度”。**

### 1.4 `hydrate`

现有逻辑里的 `hydrate` 是：

**某个节点还没准备好时，临时补全这个节点。**

它的问题是：

- 用户点击后要等
- 容易卡住
- 模型调用发生在游玩主链路里

这份重设计的核心之一，就是尽量让模板故事摆脱 hydrate。

---

## 2. 现有问题到底是什么

当前最大的设计问题不是“模型不够快”，而是：

### 2.1 两种产品模式混在了一起

现在其实同时存在两种完全不同的事情：

- **模板故事消费**
  - 用户像玩现成互动小说一样进入并选择

- **自定义故事生成**
  - 用户提供开头，系统临时生成故事

这两件事的目标天然不同：

- 模板故事追求稳定、秒开、低成本
- 自定义故事追求个性化、可生成

如果它们共用一条链路，系统就会变得很绕。

### 2.2 内容资产和用户进度没有彻底分开

现在 session 里既像在存：

- 用户游玩进度

又像在承担：

- 第一次内容生成
- 后续节点补全

这会导致：

- session 过重
- 刷新恢复复杂
- 你很难判断“这里是在读内容”还是“在生内容”

### 2.3 模型调用还在游玩主链路里

只要点一个未加载节点，系统就可能：

- 生成选项
- 生成正文
- 再写回 session

这意味着：

- 你在“玩故事”的过程中，实际上还在“等生成”

这非常影响体验。

---

## 3. 新方案的核心原则

### 原则 1：模板故事 = 内容资产

模板故事应该被看作“已经发布好的互动小说”，不是临时生成结果。

所以模板故事：

- 应该预先准备好完整 package
- 不应该在用户游玩时再调模型

### 原则 2：自定义故事 = 生成任务

自定义故事是“用户输入一个开头，请系统生产一套可玩的故事”。

所以自定义故事：

- 可以用模型
- 但模型调用应集中在“创建故事”阶段
- 不应尽量分散到每一步点击里

### 原则 3：session 只表示“玩到哪了”

session 应该轻量、明确，不再承担复杂生成职责。

它只回答：

- 你在玩哪份内容
- 你玩到哪了
- 你怎么走到这里的

### 原则 4：前端不负责生成编排

前端应该只负责：

- 进入故事
- 播放当前节点
- 点击选项
- 保存和恢复进度

而不是：

- 想办法判断什么时候要补节点
- 想办法兜底未加载状态
- 想办法和模型调用赛跑

---

## 4. 新系统拆成两种模式

## 4.1 模式 A：模板故事模式

这是书架里的模板卡片。

### 用户体验目标

- 点进就能玩
- 中途不等待模型
- 刷新也能快速恢复
- 所有用户看到的是同一套主内容

### 后端运行方式

后端预先存好完整模板 package。

用户进入时：

- 不再生成 package
- 只创建或恢复一个 session

用户点击选项时：

- 不再调模型
- 只做节点切换和状态推进

### 数据结构

模板模式要拆两层：

#### A. 模板内容表

建议叫：

- `library_story_packages`

字段建议：

- `id`
- `opening`
- `role`
- `title`
- `package_json`
- `version`
- `status`
- `created_at`
- `updated_at`

这里存的是：

**公共内容资产**

#### B. 用户游玩会话表

建议叫：

- `story_sessions`

字段建议：

- `id`
- `user_id`
- `source_type` = `library`
- `package_id`
- `current_node_id`
- `path_json`
- `state_json`
- `completed_run_json`
- `status`
- `created_at`
- `updated_at`

这里存的是：

**这个用户玩这套模板故事的进度**

### 接口设计

模板故事建议只保留这些主接口：

- `POST /api/library-stories/:packageId/start`
  - 创建或恢复该用户的 session

- `GET /api/story-sessions/:sessionId`
  - 读取当前 session 和对应 package

- `POST /api/story-sessions/:sessionId/choose`
  - 提交一个 choiceId
  - 后端更新 path / state / currentNodeId
  - 返回新的 session runtime

- `POST /api/story-sessions/:sessionId/finalize`
  - 生成 completed run

注意：

- 模板模式下不应该再有 `hydrate`

---

## 4.2 模式 B：自定义故事模式

这是用户自己输入开头的“自由创作”。

### 用户体验目标

- 允许有生成等待
- 但等待应该集中在“创建故事”时
- 进入故事后尽量顺畅

### 后端运行方式

建议把模型调用集中在“生成 package”这一步。

也就是：

1. 用户输入 opening
2. 后端调用生成链路
3. 生成一整份可玩的 package
4. 存成该用户的自定义故事 session
5. 后续游玩直接读 package

当前推荐的模型策略是：

- **自定义故事创建时全用豆包**
- 不再让自定义模式依赖 SecondMe 和豆包双模型协作

原因很简单：

- 调用链路更短
- 故障点更少
- 更容易调 prompt
- 更容易理解和维护

### 推荐生成方式

虽然是“全用豆包”，但不建议一口气让豆包直接吐出整份最终 package。

更推荐：

#### 第一步：豆包生成骨架

输出：

- title
- rootNodeId
- nodes
- 每个节点的 kind / turn / stageLabel / summary / nextNodeId

这一步只负责：

- 决定故事怎么走

#### 第二步：豆包补正文和选项

基于第一步的骨架，再让豆包输出：

- 每个节点的 scene
- paragraphs
- directorNote
- choices 文案

这一步只负责：

- 决定故事怎么写

这样虽然还是两步，但：

- 只用一个模型
- 结构校验更容易做
- 比“一次性生成完整 package”更稳

### 关键原则

自定义故事也尽量不要边玩边 hydrate。

更推荐：

- **生成一次**
- **后面只播放**

即使开始会慢一点，也比每一步点击都慢更容易接受。

### 数据结构

自定义故事不一定需要独立 package 表，第一版可以直接存到 session：

- `source_type` = `custom`
- `package_json`
- `opening`
- `role`
- `current_node_id`
- `path_json`
- `state_json`
- `completed_run_json`

以后如果自定义故事要沉淀成可分享内容，再单独抽成表。

### 接口设计

建议用一条明确的创建接口：

- `POST /api/custom-stories/generate`

请求：

- `opening`
- `role`

响应：

- 一个完整可玩的 session

后续接口和模板模式一样：

- `GET /api/story-sessions/:sessionId`
- `POST /api/story-sessions/:sessionId/choose`
- `POST /api/story-sessions/:sessionId/finalize`

也就是说：

- **区别只发生在“怎么创建第一份 package”**
- 一旦创建完，游玩接口应该统一

---

## 5. 模型应该放在哪

## 5.1 模板故事

模板故事不应该在用户点击链路里用模型。

模型如果要参与，只能在：

- 内容生产阶段
- 后台预生成阶段
- 内容入库阶段

而不是：

- 用户进入时
- 用户点选项时

### 模板故事的模型责任

- 生成候选正文
- 生成候选选项文案
- 生成结局润色

### 但这些应该发生在：

- 后台
- 离线
- 运营或开发准备阶段

不是在线游玩阶段。

## 5.2 自定义故事

自定义故事允许模型参与，但建议只放在：

- 创建故事时

当前推荐明确为：

- **结构生成：豆包**
- **选项生成：豆包**
- **正文生成：豆包**

也就是：

- 自定义故事创建时只调豆包
- 不再使用双模型协作
- 尽量避免不同模型之间的拼装复杂度

尽量不要放在：

- 继续游玩时

### 为什么不是双模型

不选“双模型协作”的主要原因是：

- 更容易卡在模型编排上
- 更难定位是结构错还是表达错
- 每次生成都要承担两套上游接口的不稳定性

所以当前更推荐的方案是：

- **模板故事：不在线用模型**
- **自定义故事：创建时全用豆包**
- **尾声签语：可继续单独实时生成**

## 5.3 尾声签语

尾声签语是附加功能，可以继续实时生成。

因为它：

- 不阻塞主游玩链路
- 可失败可重试
- 对核心体验影响没那么致命

---

## 6. 页面行为应该怎么变

## 6.1 书架页

书架页只做两件事：

- 进入模板故事
- 创建自定义故事

模板卡片状态收敛成：

- `未进入`
- `继续游玩`
- `已通关`

不再出现：

- 下载
- 预缓存
- 重新缓存
- 即将完成

## 6.2 故事页

故事页只处理：

- 展示当前节点
- 展示选项
- 切节点
- 保存
- 恢复

不再把页面设计成：

- 一边玩一边后台生内容

## 6.3 自定义入口

自由创作区要明确告诉用户：

- 这是生成模式
- 不是直接读取现成故事

文案应该强调：

- 正在生成故事
- 正在准备首幕

而不是：

- 下载
- 缓存

---

## 7. 推荐的后端接口新形态

## 模板故事

- `GET /api/library-stories`
  - 返回模板故事目录

- `POST /api/library-stories/:id/start`
  - 创建或恢复该用户 session

## 自定义故事

- `POST /api/custom-stories/generate`
  - 生成新的 custom story session

## 通用 session

- `GET /api/story-sessions/:id`
  - 获取 session + package

- `POST /api/story-sessions/:id/choose`
  - 提交一个选项，返回新的 runtime

- `POST /api/story-sessions/:id/finalize`
  - 完成故事

- `POST /api/story-sessions/:id/analyze-ending`
  - 生成尾声签语

这样会比现在这组更清楚：

- `/start`
- `/hydrate`
- `/hydrate-stream`
- `/preload`
- `/regenerate`

因为新接口一眼就能看出：

- 是模板内容
- 是自定义生成
- 还是通用 session 行为

---

## 8. 迁移建议

我建议按 4 步迁移，不要一次全重写。

### 第一步：概念分流

先在数据和接口层明确区分：

- `library`
- `custom`

哪怕第一版还没彻底重构，也先把 source type 建好。

### 第二步：模板故事去 hydrate

把模板故事改成：

- 完整 package 入库
- 用户进入只读
- 点击只切节点

这是收益最大的一步。

### 第三步：自定义故事集中生成

把自定义故事的模型调用收敛到：

- 创建故事时

而不是点击时。

### 第四步：清理旧心智

删掉这些旧概念：

- preload
- regenerate
- 下载进度
- 即将完成
- 模板故事后台补全

---

## 9. 这套方案带来的直接收益

### 对用户

- 模板故事更快
- 模板故事更稳
- 页面状态更容易理解
- 刷新恢复更自然

### 对研发

- session 逻辑更简单
- 接口职责更清楚
- 容易定位 bug
- 更容易做缓存和数据库优化

### 对产品

- 书架和自由创作的心智终于分开
- 可以解释清楚“哪里是内容库，哪里是 AI 生成”
- 后续更容易做模板运营或故事上架

---

## 10. 当前推荐结论

如果只保留一句话，我建议是：

**模板故事改成后端预生成、前端直读；自定义故事改成创建时生成、游玩时只播放。**

这会比现在“模板和自定义共用一条边玩边补的链路”清楚很多。

---

## 11. 下一步可以做什么

如果要继续推进，我建议只做下面二选一：

### 方案 A：先写正式 OpenSpec

把这份方案转成：

- proposal
- design
- tasks

适合先和同事对齐。

### 方案 B：先做最小技术验证

只验证一件事：

- 模板故事完整 package 存后端
- 用户进入模板故事时，不再触发 hydrate

适合先证明这套方向在代码上跑得通。
