## Context

这一轮反馈横跨阅读页、书架页、自定义生成链路和保存入架流程，属于典型的跨模块 UX 修复，而不是单一页面的样式微调。当前系统已经具备“正文逐段 reveal、回合结束后展示选项、书架页发起自定义生成、结局后保存故事”的基础能力，但一些状态衔接仍然过于乐观：

- 阅读页在正文 reveal 与选项滚动之间存在竞态，最后一段刚出现时就可能被选项区抢走注意力。
- 段落拆分当前会在缺少显式换段时用句号强行断句，导致引号内连续台词被拆散。
- 自由创作区只有一个开局输入框，用户无法表达“恐怖”“荒诞搞笑”这类明确预期。
- 保存书架目前走乐观更新，但没有把“已保存/已同步/不可重复保存”做成稳定状态，因此同一完成会话可能被重复入架。
- 后端入口文件仍然很长，本次实现如果再把解析、保存保护或生成状态判断写回 `main.py`，会直接违背现有分层方向。

## Goals / Non-Goals

**Goals:**
- 让每个阅读回合保持“正文先落定，再出现选项”的稳定节奏。
- 让段落拆分优先尊重原始文本结构，并尽量避免打断成对引号中的连续表达。
- 为自由创作补充轻量但明确的风格/走向输入，不把用户意图全部压缩在开局一句话里。
- 让自定义生成的等待与失败提示可区分、可恢复、可理解。
- 让保存书架对同一完成会话具备清晰反馈和幂等保护，避免重复入架。
- 把本次后端改动限制在现有 route/service/provider/parser 分层，并补齐触及模块的中文说明和 docstring。

**Non-Goals:**
- 不在本次变更中重做整个阅读页视觉风格。
- 不引入新的流式生成协议或后台任务队列。
- 不重构整个故事保存数据模型，只补强当前流程中的反馈和幂等保护。
- 不把“自由创作风格控制”扩展成复杂的多字段世界观编辑器。

## Decisions

### 阅读 reveal 与选项展示改为严格两阶段
选择把“正文 reveal 完成”与“选项 dock 可交互”拆成显式的两个前端状态，而不是继续依赖多个 watch 的自然时序碰巧对齐。这样可以保证最后一段正文至少先完成 reveal，再触发选项滚动和交互显现。

不采用“简单延时 300ms 再显示选项”的方案，因为定时器只能掩盖竞态，不能保证不同机型和不同段落数量下都稳定。

### 段落拆分从句号断句退回为‘显式换段优先 + 引号感知兜底’
后端解析层继续负责生成统一的 `paragraphs` 数组，但不再默认按每个句号切段。设计上优先顺序为：保留模型返回的空行或换行分段；若只有单段，则使用更保守的中文标点与引号感知规则做有限拆分，避免把成对引号中的一句话拆开。

不把拆分逻辑下放到阅读页，是因为段落是服务端归一化输出的一部分，前后端都依赖它，放在 parser/text 层更一致。

### 自由创作约束保持轻量输入而不是复杂表单
在书架自由创作区新增一个可选辅助输入，用来描述故事风格、情绪或希望的走向，例如“恐怖压迫感”“荒诞搞笑”“慢热拉扯”。该字段与开局正文并列提交，由后端提示词拼装时纳入语气和方向约束。

不采用固定标签多选作为唯一入口，因为用户反馈强调的是“限定想法”，文本约束对早期探索更灵活；如需标签化，可在后续独立迭代。

### 生成失败文案按失败类型归一，不再只暴露笼统网络错误
前端请求层需要把超时、中断、网络异常、服务端返回失败尽量映射为稳定的用户态错误文案；书架页则保留当前非阻塞等待体验，但在长等待时显示“仍在生成”的状态说明，并在失败后给出清晰的重试提示。

不在本次接入全局 toast 系统，避免把一次反馈修复扩成全局消息体系改造。

### 保存书架对同一完成会话实行前后端双重幂等保护
前端在成功发起保存后应立即转为“已保存”或“同步完成”状态，禁用重复点击；如果网络失败，也要明确区分“本地已保存、云端待重试”。后端若当前保存接口缺少同会话幂等约束，则应在 story save service/repository 层补一层按 `sessionId` 或等价完成态标识的去重写入。

不依赖纯前端按钮禁用来解决重复保存，因为刷新、慢网重试或多次提交仍可能产生重复记录。

### 后端改动继续沿用薄 main 约束
本次若需要调整生成请求结构、段落解析或保存去重，具体逻辑应落在 `story_text.py`、`providers/parsers.py`、保存相关 route/service/repository 模块中；`backend/app/main.py` 只允许继续做已有 wiring，不新增新的业务编排 helper。顺手补齐本次触及但未达规范的模块头部说明和中文 docstring。

不把“先做功能，后补规范”留到以后，因为这类 UX 修复会触碰多个边缘文件，现在正是顺手收敛规范的低成本时机。

## Risks / Trade-offs

- 阅读页状态拆分后，前端局部状态机会更复杂 -> 用明确布尔状态或派生状态替代隐式 watch 连锁，并补充回合切换验证。
- 更保守的段落拆分可能让少数文本段落偏长 -> 优先保证语义完整，必要时仅做有限拆分而不是激进断句。
- 自定义风格输入会让提示词更长，可能略增生成耗时 -> 约束字段保持可选且短文本，不新增多轮生成。
- 保存幂等若需要后端补强，可能触及现有存储兼容逻辑 -> 约束改动落在 repository/service 边界，避免影响入口路由装配。
- 顺手补规范会扩大少量 diff -> 只清理本次触及或明显相关的模块，不做无边界格式化。

## Migration Plan

1. 更新阅读页、书架页、自定义生成和保存同步的 OpenSpec 需求，固定验收边界。
2. 在前端拆分阅读 reveal 与选项显现状态，修正自动滚动触发点。
3. 在后端 parser/text 层收敛段落拆分规则，并同步验证阅读页展示。
4. 在书架自由创作区新增风格/走向输入，并把该字段接入自定义生成请求与提示词。
5. 在请求层与页面态中补齐长等待、超时、网络失败和可重试反馈。
6. 在保存故事流程中补齐前端成功态和后端幂等去重保护。
7. 清理本次触及但缺少说明的相关后端模块，确保不把业务逻辑新增回 `main.py`。

## Open Questions

- 保存接口当前是否已经存在基于 `sessionId` 的天然去重字段；如果没有，需要在实现时确认最稳妥的幂等键。
- 自由创作约束输入最终是单一文本框还是“文本框 + 推荐示例 chips”，可以在实现时根据现有页面空间决定，但必须保持轻量。
