## Context

当前实现默认以 OAuth 登录作为书城入口前置条件，`BookshelfPage` 在 `/api/me` 返回未认证时会直接重定向到首页；同时，尾声签语接口依赖 SecondMe access token，因此天然与登录绑定。

在生成链路上，`story_generation_service` 仍按“先生成全部 turn 选项，再生成全部节点正文”的两阶段执行。该顺序会导致选项缺乏真实正文语境，出现“选项语气、行动目标与已生成剧情不贴合”的体验问题。

此外，书城信息架构尚未显式区分公共模板内容和用户私有创作内容，用户很难建立“公共池 vs 私有宇宙”的心智。

## Goals / Non-Goals

**Goals:**
- 在不破坏现有 OAuth 安全边界的前提下引入游客模式。
- 允许游客访问公共书城与自由创作，并把游客数据限定在本地。
- 将书城拆分为 `公共` 与 `我的` 两个 Tab，明确可见性边界。
- 将自定义生成重构为“正文先行、选项后置、按回合迭代”的流程。
- 将尾声签语能力与登录状态强绑定，游客不可触发。
- 为选项点击提供一致且低干扰的音效反馈。

**Non-Goals:**
- 不在本次引入新的账号体系或游客账号上云同步。
- 不在本次改造模板故事离线播种机制。
- 不在本次扩展多人协作或跨端同步。
- 不在本次更改尾声签语分析字段结构。

## Decisions

### 1. 游客采用“前端本地身份 + 后端受限匿名上下文”

- 前端在未登录时仍可进入书城页面，但标记 `viewerMode=guest`。
- 后端新增受限的游客上下文解析方法（不从 `session` cookie 取 token），仅允许访问不依赖 SecondMe token 的能力。
- 任何依赖 SecondMe token 的接口（如尾声签语）在游客态直接返回 401/403。

选择理由：
- 保留现有 OAuth 边界（Client Secret 与 token 只在后端）。
- 避免伪造“匿名 token”带来的安全和审计歧义。

备选方案：
- 方案A：继续强制登录（放弃游客模式）。
  - 缺点：与本次需求冲突，且首访门槛不变。
- 方案B：为游客申请服务端统一 token。
  - 缺点：会混淆行为归属并放大风控风险。

### 2. 书城改为双 Tab：`公共` / `我的`

- `公共`：展示公共模板故事（所有人可见）。
- `我的`：展示当前用户可见的个人故事。
  - 登录用户：展示其云端故事 + 本地暂存故事。
  - 游客：仅展示本地生成/保存故事。
- 自由创作入口在两个模式都可用，但生成结果都落到 `我的` Tab（游客仅本地）。

选择理由：
- 与“内容来源”和“可见性边界”一致。
- 降低用户对“为什么别人看不到我作品”的认知成本。

### 3. 游客禁用尾声签语采用“双层闸门”

- UI 层：游客态不渲染“生成尾声签语”按钮。
- API 层：`/api/story/analyze-ending` 认证前置，游客请求拒绝。

选择理由：
- 单纯隐藏按钮不能防止直接调接口。
- 单纯后端拦截会导致游客在 UI 上频繁碰到报错。

### 4. 自定义生成流程改为“正文 -> 选项 -> 下一正文”

针对每个 turn 节点执行如下序列：
1. 先根据骨架与上下文生成该节点正文（scene/paragraphs/summary）。
2. 再将该正文作为强上下文输入生成该节点选项（含 tone/style/effects）。
3. 再推进到下一节点重复步骤 1-2，直至全部节点完成。

实现策略：
- 把现有 `hydrate_story_choice_nodes` 与 `hydrate_story_content_nodes` 重构为单一迭代编排器，例如 `hydrate_story_nodes_interleaved`。
- 提示词层增加显式约束：选项必须引用该节点正文中的冲突、目标或情绪落点。

选择理由：
- 选项语义可被已生成正文锚定，减少“空泛分支”。
- 与玩家阅读体验一致（先读剧情，再选行动）。

备选方案：
- 保持全局“两阶段”但在第二阶段回写修正选项。
  - 缺点：重复生成成本高、链路更复杂、可解释性弱。

### 5. 选项点击音效采用“轻量默认 + 可降级”策略

- 在选项按钮 `click` 成功受理时播放一次短促反馈音（不等待后端响应）。
- 音效资源加载失败、浏览器自动播放限制或设备静音时，交互流程必须保持可用，不阻塞选项推进。
- 音效仅绑定用户主动点击，不在自动渲染或状态恢复时触发。

选择理由：
- 强化“做选择”的即时反馈，不引入额外认知负担。
- 通过可降级策略避免把音效能力变成稳定性风险点。

## Risks / Trade-offs

- [游客本地数据丢失风险] → 在 UI 显示“游客故事仅保存在当前设备”并提示登录可同步。
- [同机多人混用导致本地故事串扰] → 本地缓存增加 guest namespace 与时间戳隔离。
- [生成时延上升] → 增加阶段化进度文案，并保留超时与重试机制。
- [旧会话兼容性] → 对历史 `story_package` 缺失字段做兼容填充，不强制迁移旧记录。
- [音效在部分设备不可播放] → 采用非阻塞播放与静默降级，确保无音效也能完整游玩。

## Migration Plan

1. 在 spec 层确认游客模式、双 Tab、流水线重构要求。
2. 后端先实现游客上下文与接口鉴权分层（不改动 OAuth 交换流程）。
3. 前端落地书城双 Tab 与游客态能力开关。
4. 重构自定义故事生成编排器为“正文先行”模式。
5. 在故事页实现选项点击音效触发与异常降级。
6. 加入回归测试：游客访问、登录访问、尾声签语按钮与接口、生成顺序一致性、音效触发时机。
7. 灰度发布并监控生成失败率与平均耗时。

Rollback:
- 若生成稳定性下降，可临时切回旧两阶段编排器（保留旧函数开关）。
- 游客模式可通过前端路由开关快速回退为“仅登录可进书城”。

## Open Questions

- 游客在 `我的` Tab 删除故事时，是否需要“回收站/撤销”机制，还是保持硬删除？
- 登录后是否需要一键导入游客本地故事到云端（本次可不做）？
- “公共 Tab”是否允许展示用户公开投稿（当前按需求先不开放，仅公共模板）？
