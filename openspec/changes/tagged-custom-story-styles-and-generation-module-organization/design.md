## Context

当前自定义故事生成已经支持附加一段风格说明，但这段说明只作为 opening 文本的一部分进入后端，两阶段节点生成时也没有被提升为独立的高优先级约束。结果是默认骨架和节点摘要往往比风格说明更有控制力，生成结果容易继续落回“默认互动小说味道”。同时，后端虽然已经形成 `routes / services / providers / repositories / domain` 的基本结构，但故事生成相关的多个根模块仍裸露在 `backend/app` 下，职责分布不够统一。

## Goals / Non-Goals

**Goals:**
- 将自由创作的主风格输入固定为单选标签，降低用户选择成本并提高生成映射稳定性。
- 确保 `言情`、`悬疑`、`恐怖`、`搞笑` 四个标签在骨架、正文、选项和结尾收束上都体现明显差异。
- 保留一个可选的补充说明字段，但只作为微调输入，不再承担主分类职责。
- 为故事生成相关模块定义更清楚的归位目标，继续减少根目录裸露模块数量。
- 继续约束 `main.py` 只承担入口装配和薄转发，不新增重量级业务编排。

**Non-Goals:**
- 不在本次变更中引入多标签、多级分类或标签推荐算法。
- 不试图一次完成整个后端目录的大重构，只覆盖故事生成直接相关模块。
- 不把自由创作扩展成复杂的世界观编辑器或长表单配置器。
- 不修改现有对外 API 的资源路径，只调整请求字段和内部模块组织。

## Decisions

### 风格输入收敛为单选标签 + 可选补充说明
自由创作入口只保留一组单选标签：`言情`、`悬疑`、`恐怖`、`搞笑`。同时保留一个可选短文本输入，用于补充时代背景、人物关系、禁忌项或期望桥段，但不再作为风格主控制项。

不采用“主风格 + 走向标签 + 自由补充”的三层模型，因为当前阶段更需要低心智负担和稳定映射，而不是更细的创作控制面板。

### 风格标签必须成为生成主约束，而不是 opening 附注
后端生成链路需要显式传递 `styleTag`，并在骨架构造、节点正文 prompt 和节点选项 prompt 中都以独立字段表达其约束。每个标签对应一套风格画像，至少定义：

- 允许的氛围词与叙事节奏
- 推荐的冲突形式
- 正文应优先体现的感官或情绪焦点
- 选项文案的默认语气分布
- 可接受的结局收束方式

不继续依赖“把风格说明拼进 opening 文本”的方案，因为这种做法在两阶段生成里权重过低，无法稳定压过骨架模板。

### 风格差异优先通过骨架族而不是单条 prompt 文案实现
四个标签应至少映射到不同的骨架族或骨架配置，而不是所有标签共用同一套“暧昧拉扯”骨架后只改几句 prompt。第一版可以允许每个标签复用部分节点数量结构，但必须让 `beat`、`pathSummary`、默认冲突类型和结局容许区间发生明确变化。

不要求每个标签都维护一整套完全独立的 100% 不共享模板，但要求用户在阅读结果时能明显感知这是不同类型而不是同一故事换词。

### 模块归位优先覆盖故事生成相邻根模块
本次目录整理只覆盖与故事生成直接相关、且当前仍裸露在 `backend/app` 根目录的模块。目标分层如下：

- `core/`: 应用级公共能力，如安全、集成清单、设置装配
- `domain/`: 稳定的故事领域常量、预置 opening、包结构规则
- `providers/`: 模型调用边界、prompt 组合、模型输出解析
- `services/`: 生成编排、运行时推进、文本归一化等业务流程

优先迁移目标：
- `story_prompts.py` -> `providers/` 下的故事 prompt 模块
- `model_providers.py` -> `providers/` 下的模型能力边界
- `openings.py` -> `domain/` 下的预置 opening 模块
- `story_text.py` -> 结合实际职责归入 `services/` 或 `providers/` 的文本归一化模块
- `integration.py` -> `core/` 下的集成清单模块

不要求这次同时迁移所有根模块，只要求新功能不再继续依赖“根目录继续长新文件”的方式。

### main.py 继续保持薄入口
本次若需要增加风格标签映射、骨架选择或 prompt 透传，优先新增/调整 service 或 provider 层函数，由 `main.py` 仅做最小 wiring。若某段逻辑超过“参数整理 + 调用 service”范畴，就不应再留在入口模块。

## Risks / Trade-offs

- 标签数量减少后，部分用户可能觉得自定义空间不够大 -> 用补充说明字段保留微调空间，并为后续扩展留出余地。
- 骨架族分流会增加测试面 -> 先固定 4 个标签，避免第一版过多组合爆炸。
- 模块迁移会带来 import 改动和潜在回归 -> 限定迁移范围，只处理故事生成直接相关模块，并保留对外 API 不变。
- 若 `story_text.py` 同时承担 provider 解析和通用文本处理，最终归位可能需要二次拆分 -> 先在 design 中接受“按职责拆成两个小模块”的可能，不强求一步到位。

## Migration Plan

1. 在 OpenSpec 中新增单选风格标签与模块归位的需求和验收场景。
2. 调整书架自由创作 UI，请求 payload 改为 `styleTag + optional note` 形态。
3. 为四个标签建立后端风格画像与骨架映射，并把 `styleTag` 显式传入节点正文/选项 prompt。
4. 校验不同标签生成出的正文和选项差异，避免仅标题或少量词汇变化。
5. 迁移故事生成直接相关的根模块到目标分层目录，并更新 import。
6. 补充本次触及模块的文件说明与中文 docstring，确保不把新逻辑重新堆回 `main.py`。

## Open Questions

- `story_text.py` 最终更适合拆到 `services/` 还是 `providers/`，需要在实现时按剩余职责确认。
- `言情` 标签第一版是否要细分为更甜或更拉扯的子风格，不在本次变更中处理。
