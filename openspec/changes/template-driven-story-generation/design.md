## Context

当前故事包流程已经实现“骨架生成 + 节点补全”的两阶段生成，但两个阶段都仍然依赖 SecondMe `act/stream`，后端主要做的是校验、重试和归一化，而不是主动控制故事结构。这导致两个直接问题：一是首次进入故事时的等待仍然偏长，因为骨架和正文都要实时生成；二是内容体验仍然受模型自由发挥影响较大，分支差异、节奏强弱和戏剧张力不够稳定。

这次改动的目标不是直接做一个完整的互动小说编辑器，而是先把现有 demo 收紧成一套更可控的运行时架构：后端提供通用高戏剧骨架模板，SecondMe 只负责当前节点选项生成，火山模型只负责当前节点正文生成，后端统一编排状态推进和结局路由。这样既保留生成式内容的弹性，也把最影响性能和体验的结构部分收回到后端。

## Goals / Non-Goals

**Goals:**
- 用后端模板替代实时骨架生成，降低故事首次进入时的等待成本。
- 固定一套偏 drama / 搞笑风格的短篇互动骨架，让故事更快进入冲突、反转和收束。
- 把选项生成和正文生成拆成两条链路，允许按任务特性分别选择模型。
- 保持现有登录、书架、story package 播放和结局分析主流程不变，只调整包的生成方式。

**Non-Goals:**
- 本次不把网站扩展成完整的互动小说编辑器或作者后台。
- 本次不追求同时支持所有题材、所有节奏模板或复杂长篇分支。
- 本次不改动 SecondMe OAuth 边界、会话机制或 MCP 对接方式。
- 本次不在设计层面承诺最终长期模型供应商，只先引入一个新的正文生成链路用于验证。

## Decisions

### 由后端提供通用高戏剧骨架模板
后端将预置一套可复用的短篇互动骨架，例如“直接出事 -> 火上浇油 -> 爆点反转 -> 最终抉择 -> 三类结局”，并通过节点意图、阶段标记和结局类型来定义故事结构。

这样做的原因是，当前问题的根源之一就是把结构性交给模型实时决定，导致生成慢且戏剧节奏不稳定。把骨架固定到后端后，系统可以先快速返回一个合法、节奏可控的 package，再按节点补全内容。

Alternative considered: 继续让模型实时生成完整骨架，只是改 prompt 或换模型。我们不采用这一方案，因为它无法根本解决结构不稳定和首屏等待重的问题。

### SecondMe `act/stream` 只负责生成节点选项
当前节点的三个可选项仍然交给 SecondMe `act/stream` 生成，但输入不再是“自由生成下一段剧情”，而是“围绕当前节点意图和选项策略输出三个有明显差异的可点击选项文案”。

这样做的原因是，选项文案既要有策略差异，又要保留一点灵活表达，仍适合交给 SecondMe。但它的职责会被限制在选项层，不再让它顺带决定整张图、结局布局或整段正文。

Alternative considered: 让后端把选项也完全模板化。我们暂不采用这一方案，因为这会让选项读感更容易僵硬，也不利于验证 SecondMe 是否适合作为“轻结构、高差异选项生成器”。

### 火山模型只负责节点正文和对白
每个节点的场景描写、角色台词和情绪表达由火山模型生成，输入材料由后端控制，包括故事开头、角色身份、节点意图、当前状态、已走路径摘要和目标戏剧效果。

这样做的原因是，正文生成更偏语言表达任务，适合切换到成本更低、速度更快的模型；同时正文只需要围绕既定节点写作，风险低于让模型决定结构。

Alternative considered: 把正文也完全预置。我们暂不采用这一方案，因为当前还处在验证阶段，先保留生成式正文更容易观察模板和模型分工是否合理。

### 后端统一负责状态推进和结局路由
后端将继续作为总控层，维护节点顺序、choice effects、relationship/persona 变化、推荐选项标记以及最终结局路由规则。模型输出必须服从既定 contract，不能自行引入新的主线结构或私自更改结局类型。

这样做的原因是，只有当后端掌握状态和路由，故事才会真正变成“结构受控、表达可变”的运行时，而不是看似分层、实则仍让模型决定流程。

Alternative considered: 让正文模型根据当前情绪和上下文自行决定结局去向。我们不采用这一方案，因为这会重新把关键结构控制权交回模型。

### 保持现有 story package 播放契约，替换 package 的生成来源
前端和已落地的 local playback 流程可以继续消费 story package，只是 package 的生成来源从“SecondMe 两阶段实时生成”改为“后端模板 + SecondMe 选项 + 火山正文”。

这样做的原因是，这能最大化复用已有书架预载、节点 hydration、本地状态推进和结局分析流程，把本次改动聚焦在生成编排层，而不是同时重做前后端播放逻辑。

Alternative considered: 为新生成架构重新设计一套全新的 package schema。我们暂不采用这一方案，因为当前 schema 已能承载节点、选项、effects 和结局，不值得在验证阶段扩大改动面。

## Risks / Trade-offs

- [单一通用模板会让不同 opening 出现套路感] -> Mitigation: 先把模板做成“高戏剧通用版”验证可行性，后续再按题材拆分多个模板。
- [SecondMe 只生成选项后，输出仍可能不够有戏或不够稳定] -> Mitigation: 把选项策略和戏剧目标写进固定输入 contract，并在后端保留去重、兜底和风格纠偏逻辑。
- [火山正文模型虽然更便宜，但文风可能不如当前默认模型稳定] -> Mitigation: 先限定它只写当前节点，不让它生成结构；同时保留本地测试脚本和配置开关用于比较输出质量。
- [双模型编排会增加配置和观测复杂度] -> Mitigation: 在后端明确拆分 provider 配置、日志标签和失败归因，避免把不同链路混在一起。
- [保留现有 package contract 可能限制模板表达] -> Mitigation: 先复用现有 contract，把复杂扩展留到模板验证通过之后再评估。

## Migration Plan

1. 在后端新增模板驱动的故事骨架构造逻辑，先并行保留现有 Two-Stage 生成实现。
2. 引入新的正文模型配置和本地测试脚本，但先不切主流程默认值。
3. 将选项生成改为围绕节点意图输出三类更有戏的选项，并让 story package 消费新输出。
4. 将正文 hydration 改为调用火山链路，并验证 package 仍能被当前前端正确消费。
5. 观察首屏等待、节点质量和故事稳定性；如果结果符合预期，再考虑移除旧的骨架生成路径。

## Open Questions

- 通用高戏剧模板是否只保留一套，还是需要同时支持 romance / comedy / suspense 等多个预设风格？
- SecondMe 选项生成是否需要显式指定底层 model，还是先沿用当前默认 provider 配置？
- 火山正文链路是否需要严格 JSON 输出约束，还是只要求返回正文/摘要/局势提示这几个固定字段？
