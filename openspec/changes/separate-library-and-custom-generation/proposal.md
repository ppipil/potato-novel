## Why

当前互动小说系统把模板故事消费、自定义故事生成、节点补全和缓存状态混在同一条链路里，导致几个持续问题：

- 模板故事进入和游玩仍然可能等待运行时生成
- `hydrate` 逻辑复杂且容易卡住
- 用户很难理解“这是在加载现成内容，还是在现场生成新内容”
- session 同时承担内容生成和进度保存职责，系统难以维护

我们需要把“模板故事”和“自定义故事”彻底分开建模：

- 模板故事回归为后端预生成、前端直读的固定内容
- 自定义故事保留生成能力，但只在“创建故事”时调用模型

## What Changes

- 将模板故事定义为数据库中的完整固定 story package，所有用户共享同一套内容底稿。
- 模板故事的进入与游玩不再依赖运行时 `hydrate`；用户点击选项时只推进 session，不再现场生成节点。
- 将自定义故事定义为独立的生成模式，仅在用户提交 opening 时调用豆包生成完整可玩 package。
- 自定义故事创建时统一使用豆包，不再采用 SecondMe + 豆包双模型协作。
- 明确拆分“内容资产”和“游玩进度”：
  - 模板故事 package 是公共资产
  - session 只记录用户玩到哪、选了什么、当前状态是什么
- 收敛前端心智：
  - 模板故事是“进入 / 继续游玩”
  - 自定义故事是“生成 / 进入”
  - 不再混用下载、预缓存、重新缓存、后台补全等文案

## Capabilities

### New Capabilities

- `library-story-catalog`: 管理和分发后端预生成的模板故事 package。
- `custom-doubao-story-generation`: 定义自定义故事创建时的豆包生成链路。
- `story-session-playback`: 定义模板与自定义故事在生成完成后的统一游玩会话与节点推进行为。

### Removed From Scope

- `ai-persona-guidance` 不在本次变更范围内。

## Impact

- 影响后端故事数据模型，需要区分模板内容资产和用户 session。
- 影响模板故事进入接口，需要从“生成/补全”改为“读取/开始”。
- 影响自定义故事生成接口，需要明确改成“创建时全用豆包”。
- 影响前端书架、故事页和自定义入口的文案与状态机。
- 影响现有 `preload` / `regenerate` / `hydrate` 相关逻辑的职责边界，其中模板故事路径应逐步移除这些依赖。
