## Why

当前互动小说链路把“公共内容分发”和“用户个性化生成”混在一起处理，导致书城模板故事也要经历骨架生成、节点补全和预热状态管理，进入慢、状态文案混乱、用户难以理解“什么时候是在加载，什么时候是在生成”。现在需要把固定内容消费和自由创作生产拆开，让书城故事回到稳定可直读的内容库，自由创作才承担生成成本和阶段反馈。

## What Changes

- 将书城/模板故事改为数据库中的固定 story package，所有用户看到同一套故事底稿，进入时直接加载，不再为每个用户重新生成骨架和正文。
- 将自由创作单独定义为“生成链路”，仅在用户输入自定义开头时调用 SecondMe 生成故事骨架、开场和首层分支，并提供真实阶段反馈与预计耗时说明。
- 将用户游玩过程中的 SecondMe 职责收缩为“分身选择倾向判断”和个性化解读，而不是在每次进入书城故事时重新生成整本互动包。
- 调整书架与书城 UI 心智：书城卡片展示“加载/进入”状态，自由创作展示“生成”状态，不再混用“下载/预缓存”文案。
- 为后续性能优化奠定边界：书城故事追求秒开，自由创作追求分阶段可感知与首幕可进入。

## Capabilities

### New Capabilities
- `library-story-catalog`: 管理和分发数据库中的固定书城故事包，并定义书城故事的加载与进入行为。
- `custom-story-generation`: 定义自由创作的生成流程、阶段状态、预计时长提示，以及“首幕可进入”的生成门槛。
- `ai-persona-guidance`: 定义用户游玩过程中的分身选择倾向判断、推荐解释和个性化尾声解读能力。

### Modified Capabilities
- None.

## Impact

- 影响后端故事模型、书城读取接口、自由创作生成接口和运行时状态管理。
- 影响前端书架、书城、自由创作入口和状态文案设计。
- 影响当前 `preload` / `hydrate` 的职责边界，以及数据库中 story package 的来源分类。
- 影响 SecondMe API 使用方式：书城读取减少运行时生成，自由创作保留内容生成，游玩过程增加轻量人格判断调用。
